ТРАНСКРИБАЦИЯ ВСТРЕЧИ: Разработка системы управления задачами и командой
Дата: 20.03.2026
Участники:
  Заказчик: Екатерина Романовна Белова, операционный директор, ООО «СтройГрупп»
  Аналитик-разработчик: Павел Сергеевич Орлов

---

Павел: Екатерина Романовна, добрый день. Расскажите о задаче — что именно вас не устраивает в текущих инструментах и что хотите получить?

Екатерина: Добрый день, Павел. У нас строительная компания, 200 сотрудников, несколько объектов одновременно. Сейчас используем Excel и мессенджеры для управления задачами. Это хаос. Задачи теряются, непонятно кто за что отвечает, нет контроля сроков. Хотим нормальную систему управления проектами, заточенную под строительство.

Павел: Понятно. Смотрели готовые решения — Jira, Asana, Битрикс24?

Екатерина: Смотрели. Битрикс24 слишком перегружен и дорог для нас. Jira — для IT, не для строителей. Asana близко, но нет специфики строительства: нет привязки к объектам, нет работы с нарядами, нет интеграции с нашей сметной программой.

Павел: Значит, нужна кастомная разработка. Давайте разберём функциональность. Что обязательно должно быть?

Екатерина: Первое — иерархия: компания → объект → этап → задача. Второе — назначение ответственных, дедлайны, статусы. Третье — наряды на работу: прораб создаёт наряд, рабочие отмечают выполнение. Четвёртое — фотоотчёты прямо из приложения — прораб фотографирует, прикрепляет к задаче. Пятое — уведомления: просроченные задачи, изменения статусов.

Павел: Мобильное приложение обязательно? Прорабы работают на объектах.

Екатерина: Да, это критично. iOS и Android. Причём должно работать при плохом интернете — на объектах бывают проблемы со связью.

Павел: Значит, нужен офлайн-режим с синхронизацией. Это усложняет разработку, но реализуемо. Интеграция со сметной программой — это что за программа?

Екатерина: Гранд-Смета. Нужно импортировать сметы оттуда и привязывать к объектам и этапам.

Павел: Гранд-Смета экспортирует в XML или Excel — можем сделать импорт через файл на первом этапе, а потом API-интеграцию если они предоставят.

Екатерина: Хорошо, файловый импорт на старте нас устроит.

Павел: Роли пользователей — какие нужны?

Екатерина: Администратор — полный доступ. Руководитель проекта — управление объектом. Прораб — создание нарядов, задач на своём объекте. Рабочий — только просмотр своих нарядов и отметка выполнения. Бухгалтер — просмотр нарядов для расчёта зарплаты.

Павел: Отчётность нужна?

Екатерина: Да. Отчёт по выполнению задач за период, отчёт по нарядам для расчёта зарплаты, сводка по объекту — что сделано, что в работе, что просрочено.

Павел: По технологиям — есть предпочтения?

Екатерина: Нет, доверяем вашей экспертизе. Главное — надёжность и возможность развивать систему.

Павел: Предложу: веб — React + TypeScript, мобильное — React Native (один код для iOS и Android), бэкенд — Python FastAPI, база — PostgreSQL, офлайн-синхронизация — через очередь сообщений.

Екатерина: Звучит хорошо. По срокам — нам нужно запустить до начала строительного сезона, то есть к 1 мая.

Павел: Сейчас конец марта. Полтора месяца — это очень мало для такого объёма. Реалистично за это время сделать веб-версию с базовой функциональностью: иерархия объектов, задачи, наряды, роли. Мобильное приложение и офлайн-режим — следующий этап, июнь-июль.

Екатерина: Понимаю. Давайте так. Главное — чтобы прорабы могли работать с нарядами через телефон хотя бы в онлайн-режиме к маю.

Павел: Тогда сделаем адаптивный веб — он будет работать на телефоне через браузер. Нативное приложение — второй этап.

Екатерина: Договорились. Бюджет на первый этап — 800 тысяч рублей.

Павел: Принято. Риски: сжатые сроки могут повлиять на качество тестирования. Рекомендую заложить 2 недели на UAT с вашей стороны. Также интеграция с Гранд-Сметой зависит от формата их экспорта — нужно получить примеры файлов заранее.

Екатерина: Хорошо, пришлю примеры сегодня. Когда сможете начать?

Павел: Со следующей недели. К пятнице пришлю ТЗ и план спринтов.

Екатерина: Отлично. Спасибо, Павел.

Павел: Спасибо, Екатерина Романовна. До связи.
